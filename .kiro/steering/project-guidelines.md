# Project Guidelines - Alpaca Real-time Alert System

## 아키텍처 원칙
- Kafka + Redis 기반 실시간 스트리밍 파이프라인 구성
- 최소한의 리소스 사용 (경량 컨테이너, 메모리 효율적 설계)
- 알림 채널: ntfy.sh 기반 iOS 푸시 알림
- 전일 종가는 SQLite에 영속 저장 (매일 갱신)

## 코드 작성 규칙
- 모든 코드에 상세한 한글 주석 작성
- 함수/클래스/모듈 단위로 목적과 동작 방식 설명
- 복잡한 로직에는 단계별 주석 포함
- 과도하게 단순화하거나 장황한 코드 지양, 가독성 우선

## 문서화
- README.md에 전체 시스템 스펙과 아키텍처 도식(Mermaid) 포함
- 각 컴포넌트의 역할과 데이터 흐름 명시

## 리소스 최적화
- 불필요한 폴링 금지, WebSocket 기반 실시간 수신
- Redis는 캐시 + 상태 저장(쿨다운/레이트리밋) 용도로만 사용
- Kafka 파티션은 최소 구성 (필요 시 확장)
- 컨테이너 메모리 제한 설정 필수

## 테스트 정책
- 복잡한 테스트 코드나 다수의 단위 테스트 파일 생성 금지
- 각 모듈 구현 시 개별 단위 테스트를 매번 작성하지 않는다
- 검증은 가벼운 스키마/인터페이스/동작유무 체크만 수행
  - 함수가 올바른 타입을 반환하는지
  - 필수 필드가 존재하는지
  - 에러 시 올바른 예외 타입이 발생하는지
- 세부 로직(분기, 계산, 엣지케이스) 단위 테스트는 작성하지 않는다
- 외부 API/서비스는 초미니 더미 데이터(최대 2~3개)로 mock/stub 대체
- 검증 타이밍: 작업 단위가 끝난 후 한 번만 통합 검증 (import/타입 에러 확인)

## 커밋 메시지
- 간략하게 요약한 한 줄 위주로 작성
