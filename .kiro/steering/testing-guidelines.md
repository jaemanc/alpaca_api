# Testing & Development Guidelines

## 테스트 최소화 원칙
- 코드 작성시 복잡한 테스트코드는 생성하지 않는다.
- 각 컴포넌트(모듈) 구현 시 개별 단위 테스트를 매번 작성하지 않는다
- 테스트는 작업 단위(wave)의 마지막에 한 번만 통합적으로 검증한다
- 검증 방식: 로직 테스트 대신 **가벼운 스키마/인터페이스/동작유무 체크**만 수행
  - 함수가 올바른 타입을 반환하는지
  - 필수 필드가 존재하는지
  - 에러 시 올바른 예외 타입이 발생하는지
- 세부 로직(분기, 계산, 엣지케이스)에 대한 단위 테스트는 작성하지 않는다

## API/외부 서비스 테스트 시 더미 데이터 사용
- 개발/테스트 단계에서 외부 API나 서비스 호출 시 **초미니 더미 데이터**를 사용한다
- 더미 데이터 예시: 1~2개의 최소 필드만 포함한 fixture
- 실제 네트워크 호출 없이 mock/stub으로 대체
- 대량 데이터 fixture 금지 (최대 2~3개 항목)

## 테스트 작성 시 금지 사항
- 하나의 모듈에 20개 이상의 테스트 케이스 작성 금지
- 동일한 로직을 다른 입력값으로 반복 테스트하는 것 금지
- 테스트 파일이 200줄을 초과하지 않도록 한다

## 검증 타이밍
- 구현 태스크 완료 후 바로 테스트를 작성하지 않는다
- wave 단위로 모든 구현이 끝난 후 `pytest`를 한 번 실행하여 import 에러, 타입 에러만 확인한다
- Checkpoint 태스크(13, 17)에서만 전체 테스트 스위트를 실행한다
